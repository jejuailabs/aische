## 1. 프로젝트 한 줄 정의

**목표(Goal)를 계층형 프로젝트 데이터(만다라트 Node)로 구조화하고, 이를 캘린더·투두·간트·로그와 하나의 데이터 모델로 통합 관리하며, 음성 AI 비서를 통해 자연어로 조작할 수 있는 개인 생산성 플랫폼.**

- 만다라트/트리/리스트/캘린더/간트/로그는 기능이 각각 다른 게 아니라 **동일한 데이터의 다른 View**다.
- 입력은 텍스트, 음성, 파일, Gmail, 카카오톡 공유하기 등 다양한 채널에서 들어오지만 모두 **하나의 파서를 거쳐 동일한 데이터 모델(Node)로 귀결**된다.

## 2. 핵심 설계 원칙 (반드시 지킬 것)

1. **단일 데이터 모델(Single Source of Truth)**
   모든 View(만다라트/캘린더/투두/간트)는 동일한 Node 컬렉션을 참조한다. 특정 View에서 수정하면 모든 View가 즉시 동기화된다.

2. **AI 산출물은 반드시 Firebase에 저장한다.**
   AI가 생성하는 모든 결과물(생성된 일정 초안, 아바타/배경 이미지, 파싱 결과, 요약, 회고 텍스트 등)은 **로컬 파일이나 배포 번들에 포함시키지 않고 전부 Firebase(Firestore/Storage)에 저장**한다. 정적 에셋으로 빌드에 번들링하는 것은 금지.

3. **Draft → Confirm 패턴**
   AI가 자동으로 채우거나 배치하는 모든 항목(일정 필드 부족 시 되묻기, 프로젝트 자동 매칭 등)은 바로 확정되지 않고 **초안(Draft) 상태로 생성 → 사용자 확인 후 확정(Confirm)**되는 흐름을 따른다.

4. **입력 소스 추상화 (확장성 원칙)**
   음성/텍스트/파일/Gmail/카톡/문자 등 모든 입력은 `InputSource`라는 추상 인터페이스를 통해 처리 로직에 들어온다. 특정 플랫폼 API에 비즈니스 로직이 종속되지 않도록 하여, 추후 채널이 추가되거나(예: 문자 연동을 위한 네이티브 쉘 추가) 교체되어도 핵심 로직은 재사용된다.

5. **플랫폼 확장성**
   현재는 PWA(웹) 단일 플랫폼으로 시작하지만, 추후 Capacitor 등으로 네이티브 쉘을 씌워 OS 레벨 권한(SMS 읽기 등)이 필요한 기능을 확장할 수 있도록 프론트엔드/비즈니스 로직을 분리해서 설계한다.

## 3. 시스템 구성 요소 (모듈 개요)

- **입력 계층**: 텍스트 챗, 음성(STT/TTS), 파일 첨부, Gmail 연동, 카카오톡 공유하기
- **AI 오케스트레이션 계층**: 자연어 파싱 → 필수 필드 검증 → 부족 시 되묻기 → Draft 생성 → 사용자 Confirm → Node 저장 / 프로젝트 자동 매칭
- **데이터 계층**: Firestore 기반 통합 Node 모델 (`data-model.md` 참조)
- **View 계층**: 만다라트(트리), 캘린더(월/일), 투두 리스트, 간트, 로그, 통계
- **UI 셸**: PC 대시보드(위젯 워크스페이스), 모바일(하단탭바+스와이프)
- **부가 기능**: 알림(최대 5단계), 다크/라이트 모드, 다국어, 관리자 모드, 커스텀 아바타/배경

## 4. 페이즈 계획

전체 기능을 다 구현하되 아래 순서로 단계화한다. 각 페이즈는 이전 페이즈 위에 누적된다.

| Phase | 내용 |
|---|---|
| **Phase 1** | Firebase Auth(구글 로그인) + Firestore 데이터 모델(Node 통합 스키마) + 캘린더(월간 미니타임라인 + 일간 상세) + 투두 기본 CRUD + 다크/라이트 모드·다국어 뼈대 |
| **Phase 2** | AI 텍스트 입력(챗 기반 일정/투두 등록, 필수 필드 검증, Draft→Confirm) + 파일 첨부 파싱 |
| **Phase 3** | 만다라트/프로젝트 트리(무한확장 노드, 다중 View 동기화) + 캘린더·투두 항목의 AI 자동 프로젝트 매칭(Draft→Confirm) + 미분류함/병합 UI |
| **Phase 4** | 음성 인터페이스(실시간 STT/TTS, 파형 반응 원형 비주얼라이저) |
| **Phase 5** | 외부 연동 — Gmail(OAuth, 일정성 메일 파싱), 카카오톡 "공유하기"로 텍스트 인입. 문자(SMS) 연동은 이 페이즈 이후 네이티브 쉘 필요 시 별도 확장 항목으로 보류 |
| **Phase 6** | 관리자 모드, 알림 세부 설정(시/일/주/월, 최대 5개), 로그/통계/시각화, 커스텀 아바타·배경(Lottie/Rive 포함 여지) 고도화 |

## 5. 데이터 흐름 요약 (End-to-End)

```
[입력: 음성/텍스트/파일/Gmail/카톡공유]
        ↓ InputSource 어댑터
[AI 파서] → 필수 필드(날짜·시간·소요시간·대상자·장소·내용) 검증
        ↓ 부족하면 사용자에게 되묻기 (대화형)
[Draft Node 생성] (Firestore에 draft 상태로 저장)
        ↓ 사용자 Confirm
[Confirmed Node] → Firestore 정식 컬렉션 반영
        ↓ AI 프로젝트 매칭 시도
   ├─ 매칭 성공 → 해당 만다라트 프로젝트 하위에 배치 제안 (Draft) → 사용자 승인
   └─ 매칭 실패 → "미분류함"으로 이동, 사용자가 직접 프로젝트 생성/병합
        ↓
[모든 View(캘린더/투두/만다라트/간트/로그)에 실시간 반영]
```

## 6. 미결정/추후 논의 필요 항목

- 투두 모듈 기능 상세 — 시장 벤치마킹(Todoist, TickTick, Things3 등) 아직 미실행, `todo.md`에서 진행
- 문자(SMS) 연동의 최종 구현 방식(네이티브 쉘 도입 시점)
- AI 이미지 생성(아바타/배경) API 비용 구조 및 갤러리 방식 상세
- Lottie/Rive를 활용한 아바타 애니메이션 도입 여부 및 시점 (일단 스펙에는 포함, 구현 우선순위는 후순위)
