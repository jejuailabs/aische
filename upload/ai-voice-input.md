# ai-voice-input.md — AI 대화 / 음성 인터페이스 / 다중 입력 소스

## 1. 컨셉

"자비스(Jarvis)"형 AI 비서. 텍스트/음성/파일/외부 채널을 통해 자연어로 일정·투두·목표를 등록하면, AI가 파싱하여 Draft를 만들고 사용자 Confirm을 받아 정식 등록한다.

## 2. 입력 채널 (InputSource)

`core.md`의 "입력 소스 추상화" 원칙에 따라 모든 채널은 공통 인터페이스로 처리한다.

```ts
interface InputSource {
  channel: "voice" | "text" | "file" | "gmail" | "kakao_share" | "sms";
  ingest(raw: unknown): Promise<ParsedIntent>;
}
```

| 채널 | 방식 | Phase |
|---|---|---|
| 텍스트 | 챗 UI 직접 입력 | Phase 2 |
| 파일 첨부 | 문서/이미지 업로드 → 내용에서 일정/태스크 추출 | Phase 2 |
| 음성 | 마이크 버튼 → 실시간 STT → 파싱 → TTS 응답 | Phase 4 |
| Gmail | OAuth 연동, 일정성 메일(예약 확인/초대장 등)에서 자동 추출 | Phase 5 |
| 카카오톡 | 사용자가 대화를 앱으로 "공유하기"로 전달 → 텍스트 파싱 (카톡 API로 대화 자동 수집은 정책상 불가하여 공유하기 방식만 채택) | Phase 5 |
| 문자(SMS) | OS 레벨 권한 필요 → 웹(PWA)에서는 구현 불가. 네이티브 쉘(Capacitor) 도입 시 후순위로 확장 검토 | 보류 |

## 3. 음성 인터페이스 UX

- 화면 중앙 원형 아바타 (`design-system.md`의 커스텀 아바타 시스템과 연결)
- 마이크 버튼 탭 → 음성 입력 시작
- 음성 입력 중, 원형 아바타 주변에 **입력 주파수에 반응하는 파형/원형 애니메이션**이 표시됨 (오디오 레벨에 따라 원이 확장/수축)
- STT → 파싱 → (필요 시) TTS로 되묻기 → 사용자 음성/텍스트 응답 → 최종 Confirm까지 대화형으로 진행
- 실시간 스트리밍 STT/TTS가 필요하며 지연시간·비용이 발생하는 영역이므로, 초기엔 발화 종료 후 일괄 처리 → 추후 실시간 스트리밍으로 고도화 (Phase 4 내 세부 단계)

## 4. 자연어 파싱 → 필수 필드 검증

모든 채널의 입력은 다음 필수 필드를 채워야 정식 일정/투두로 등록된다 (`calendar.md` §5와 동일 기준):

- 날짜
- 시간 (몇 시부터 몇 시간)
- 대상자
- 장소
- 내용

**부족한 필드가 있으면 AI가 사용자에게 되묻는다.** (텍스트 채널이면 챗으로, 음성 채널이면 TTS로) 사용자가 답변하면 필드를 채우고, 최종적으로 요약을 보여준 뒤 **컨펌을 받아야 등록**된다.

## 5. Draft → Confirm 흐름 (상세)

1. 입력 수신 → `AiMeta.status = "draft"`로 Node 임시 생성 (Firestore에 저장, 로컬 저장 금지)
2. 필수 필드 부족 시 `clarificationLog`에 대화 기록을 남기며 되묻기 반복
3. 필드 완성 → 사용자에게 요약 제시("6/23 오전 10시, 부모님과 해녀체험, 2시간, 협재해수욕장 — 등록할까요?")
4. 사용자 Confirm → `status: "confirmed"`로 전환, 정식 View에 노출
5. 동시에 AI가 프로젝트 자동 매칭 시도 → 매칭 제안도 Draft로 별도 생성 → 사용자 승인 (`mandarat.md` §16)

## 6. AI 목표 확장 인터페이스

만다라트 노드에서 직접 AI 호출 가능 (`mandarat.md` §15): 세부 목표 생성, 실행 계획 생성, 일정 자동 생성, 예상 기간 계산, 위험 요소 분석, 우선순위 추천, 회고 작성. 모두 기존 프로젝트 Node를 기반으로 동작하며 산출물은 Draft로 생성된다.

## 7. AI 산출물 저장 원칙

모든 AI 생성 결과(파싱된 일정 초안, 요약 텍스트, 이미지 등)는 **Firebase(Firestore/Storage)에만 저장**하며 로컬 파일 시스템이나 배포 빌드에 포함하지 않는다 (`core.md` §2-2).
